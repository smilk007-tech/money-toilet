# 돈버는 화장실 — AI 코딩 가이드

> **이 파일은 설계가 변경될 때마다 반드시 함께 수정한다.**
> 미래 AI 세션이 이 파일만 보고 설계 의도를 완전히 파악할 수 있어야 한다.

---

## 이 서비스가 뭔지

근무 중 화장실에 숨어서 변기 위에 앉아있는 시간만큼 급여가 실시간으로 쌓이는 웹앱.
타겟은 **20~50대 한국 직장인**. 점심 전 화장실 10분, 오후 슬럼프 때 잠깐 도피하는 그 시간.
실제 돈은 없다. **그냥 변기 위에 앉아서 시간 때우고 낄낄거리고 끝.** 이게 전부다.

### 성공 기준
- 하루 1,000 방문 · 인당 체류 1분 이상 · 공유하기가 많이 일어나는 것
- 바이럴이 먼저다. 광고·후원·수익화는 잘 터진 다음의 얘기다
- 지금은 순수 바이럴 기계로만 설계하면 된다

### 인프라 제약 (절대 잊지 말 것)
| 서비스 | 역할 | 비용 |
|--------|------|------|
| Vercel | Next.js 프론트 · API routes | 무료 플랜 |
| Railway | Node.js + Socket.io 채팅 서버 | $5/월 기본 |
| Upstash Redis | 세션 · 누적 · 어드민 설정 | 무료 플랜 |

Railway $5 + Upstash 무료로 **모든 것을 커버해야 한다.**
기능 추가·변경 시 항상 서버 부하와 Redis 커맨드 수를 먼저 따져라.
B급 감성의 서비스지만 코드와 인프라는 최적화를 끝까지 깎아라.

## 핵심 설계 철학

### 1. 물 흐르듯 자연스러움

**이 서비스의 모든 요소는 의도다. 우연처럼 보여야 한다.**

사용자는 "이거 프로그램이 일정하게 올려주는 거잖아"를 느끼는 순간 몰입이 깨진다.
텍스트 색깔, 레이아웃 계층, 투명도, 노출 타이밍, 딜레이, 애니메이션 — 전부 계산된 것이지만 계산처럼 보이면 안 된다.

코드를 짜기 전에 항상 이 질문을 먼저 해라:
> "이걸 실제 사람이 만들었다면 이렇게 똑같이 반복되겠냐?"

### 2. 소음 최소 원칙 (Minimum Noise)

**사용자에게 필요하지 않은 정보는 보여주지 않는다.**

- UI 요소는 사용자가 그것을 받아들일 준비가 됐을 때 등장한다 (진입 즉시 다 보여주지 않음)
- 경고·안내·차단은 현재 맥락 안에서 조용히 처리한다. 팝업·모달로 흐름을 끊지 않는다
- 연결 끊김도 즉시 외치지 않는다. 짧은 블립은 아무것도 안 보여주고 넘긴다
- 정보는 순서가 있다: 월급 설정 → 돈 쌓임 → 물내림 → 영수증 → 공유. 이 순서를 역행하거나 건너뛰지 않는다

### 3. 익명성 + 휘발성

이 앱은 로그인이 없고, 내 돈이 어딘가에 영구 저장되지 않는다. 그게 의도다.

- **익명**: 닉네임은 랜덤 생성 (성씨 + 형용사 + 직급). 실제 이름을 쓸 필요도 없고, 쓰라고 유도하지도 않는다
- **휘발성**: 번 돈은 물내리면 명세서에 기록되지만 어딘가 "쌓이는" 건 아니다. 화장실에서 나가면 끝
- **익명 소셜**: 옆 칸 말풍선은 이름 없이 상황만 공유한다 ("N원 벌고 물내림 ㅋㅋ")
- **결과적 효과**: 사용자가 부담 없이 행동할 수 있다. 계정이 없으니 뭘 해도 기록이 안 남는다는 해방감

### 4. 개그 코드 — 스멀스멀 스며드는 B급

이 서비스의 유머는 직접 표현하지 않는다. **상황이 개그다.**

- 가장 좋은 예: "급여명세서". 실제 회사 급여명세서처럼 생긴 양식에 적힌 항목이 "물내림 1회 / 3,240원". 이 괴리가 웃음 포인트다. 설명할 필요가 없다.
- 닉네임 설계가 이 철학을 가장 잘 보여준다: `NICK_FRONT_WORDS`는 "못끊는", "못참는", "많이모은" 같은 형용사다. 화장실 상황을 직접 언급하지 않지만, 맥락을 알면 "아 그 사람 ㅋㅋ"가 자동으로 연상된다.
- 말풍선 멘트도 같다: "N원 벌고 물내림 ㅋㅋ 짝짝" — "물내림"이라는 단어 하나로 상황을 함축하고, "짝짝"으로 박수를 쳐준다. 자축(self-congratulation)의 개그.

**절대 하면 안 되는 것**: 저질 화장실 유머. "똥", "싸다", "지리다" 같은 말초 자극은 이 서비스에 없다. 이런 단어가 들어가는 순간 B급이 아니라 그냥 불쾌해진다.

개그는 **상황이 만들어야 한다.** 텍스트가 설명하면 이미 늦었다.

### 5. 의문을 품게 하면 안 된다

이 서비스의 개그 코드는 **전제를 진지하게 받아들이는 것**에서 나온다.
사용자가 "왜?", "이게 실제로 가능해?", "이게 말이 돼?"를 생각하기 시작하면 개그가 사라진다.

UX에서 의문이 생기는 순간은 두 가지다:
1. 기계적인 패턴을 발견했을 때 ("아 이거 N초마다 자동으로 올라가는 거네")
2. 뭔가 이상할 때 ("왜 갑자기 숫자가 확 바뀌지?", "왜 버튼이 갑자기 생겼지?")

두 경우 모두 자연스럽게 처리해야 한다. 패턴은 랜덤화하고, 등장은 부드럽게 한다.

## 타겟 페르소나

- **직군**: 사무직, 공장직, 서비스직 등 정해진 근무시간이 있는 한국 직장인
- **연령**: 20~50대
- **상황**: 오전 집중근무 중 잠깐 도피 / 오후 슬럼프 / 점심 먹고 소화시키며 / 퇴근 전 눈치 보며
- **디바이스**: 99% 모바일. 손가락 하나로 조작한다고 전제한다
- **심리**: "나만 이러는 거 아니야" / "이 시간도 내 시간이지" / 죄책감 + 쾌감 혼재
- **공유 동기**: 자랑하고 싶은데 대놓고는 못 하는 심리 → 병맛 코드로 포장

## 넛지 레이어 전체 지도

모든 넛지는 사용자를 **조용히** 다음 행동으로 이끈다. 강요하지 않는다.

### 입장 직후 — 적응 시간

| 상태 | 유예 시간 | 이유 |
|------|-----------|------|
| 완전 첫 방문 | 15초 | 화면이 뭔지 파악할 시간 |
| 조작 후 이탈 | 10초 | 기억을 되살릴 시간 |
| 모든 넛지 완수 | 5초 | 익숙하니까 짧게 |

- 진입 후 첫 **3초**(`ADAPTATION_TIME`): 물내림 버튼 비활성. 씬부터 흡수하게.
- 소켓 연결 직후 **7초**(`graceMs`): 봇 메시지·타인 말풍선 억제. 연결 직후 채팅창이 쏟아지는 이질감 방지.
- 첫 방문자에게만 **3.2초 후**(`WELCOME_DELAY_MS`) 옆 칸 말풍선 1개. 첫인상이 썰렁하지 않게, 즉시 뜨면 기계적이므로.

### 월급 설정 유도

- 진입 **10초 후** 월급 버튼 미클릭 시 `👈` 배지 등장 → 한 번 누르면 영영 사라짐
- 슬라이더 조작 즉시 "1초에 N원 버는중" 반영 → 월급을 초당 수입으로 번역해 체감 극대화

### 정보 단계적 노출

- **"오늘 다같이 N원" HUD**: 서버 첫 연결 후에야 등장. 처음부터 보이면 노이즈.
- **"총 N원" 영역**: 첫 번째 물내림 후 **5초 뒤** 슬라이드인. "행동 → 새 UI 등장" 성취감.
- **영수증 버튼**: 두 번째 물내림 후 **5초 뒤** 최초 등장. 첫 물내림은 적응 단계.
  - 이후 물내림마다 🧾 아이콘 흔들림 (5초 딜레이)
  - 미클릭 상태면 `👉` 배지 추가로 주의 끌기

### 물내림 — 핵심 행동

- **5초 쿨다운**: 연타 방지. 버튼 `disabled` + `flush--cooldown` 시각화.
- **60분 동결**: 타이머가 60분을 꽉 채우면 적립 동결 + 버튼 강조 (`flush-btn--urgent`). 강제 물내림이 아니라 시각적 압박으로 수동 유도.
  - 60분 달성 멘트 (`CONSTIPATION_LINES`): "1시간 동안 N원 벌고 물내림ㅋㅋ"
  - 더 화려한 컨페티 (`CONFETTI_EMOJIS_CAPPED`)
- **부재중 자동 적립**: 새로고침 시 자리 비운 시간 크레딧 (물내림 1회 이상 경험자만 — 뉴비가 MAX 먹는 거 방지)
- **`REST_LINES`**: 월급 0원 상태에서 물내리려 하면 "지금 비우면 손해!" 등으로 자연스럽게 만류

### 급여명세서 도장 흐름

공유로 가는 길을 "내가 직접 확인했다"는 행위로 열게 한다. 강요가 아니라 의식(ritual).

```
물내림 → 모달 열림
  → "👉 위 내용이 사실임을 확인합니다" 버튼 표시
     (이 동안 "내 월급은 공개되지 않습니다" 문구는 opacity:0 — 공간 차지, 안 보임)
  → 버튼 클릭 (내가 선택한 행위) → 0.5초 기대감
  → 도장 쾅 + 효과음 (보상 순간)
  → 저장/자랑하기 버튼 위로 올라옴 (0.38s)
  → "내 월급은 공개되지 않습니다" 뒤이어 올라옴 (0.38s 딜레이 + 0.32s)
```

- 도장 텍스트: 항상 "지급완료 / 돈버는화장실" — **티어/상위N% 개념 없음, 절대 되살리지 말 것**
- 재열람 시: 즉시 `done` 상태 (이미 찍은 도장을 다시 안 찍음)
- 레이아웃 시프트 0: `opacity:0`으로 공간 유지 후 페이드인

### 공유 → 바이럴

- OG 이미지: 검은 배경에 금액만 크게 → 숫자가 궁금해서 링크 클릭
- 공유 페이지: 영수증 + 실시간 슬롯머신 배너 + "{닉}님처럼 나도 벌기" CTA
- CTA 클릭: SPA 라우팅으로 메인 이동 (리로드 없음, 같은 vid 재연결)

## 소셜 연출 — 공간 은유와 말풍선

이 앱의 레이아웃은 **공중화장실 1인칭 시점**을 그대로 구현한 공간이다.
나는 가운데 칸에 있고, 왼쪽과 오른쪽에 이웃 칸이 있다. 모든 UI 위치가 이 공간 은유에서 나온다.

- **좌/우 벽 말풍선**: 타인의 채팅·물내림 멘트. 왼쪽 칸, 오른쪽 칸 사람이 하는 말이다.
- **가운데 myFeed**: 내 채팅 말풍선은 가운데 컬럼에 등장. 내가 가운데 칸에 앉아있으니까.
- **휴지 걸이는 왼쪽 벽**: 화장실 휴지는 보통 왼쪽에 걸려있다. 이스터에그가 발동되면 왼쪽 말풍선이 반응한다. 휴지가 왼쪽에 있으니 왼쪽 칸 사람이 눈치채는 것이 자연스럽고, 동시에 사용자가 방금 뜯은 왼쪽 방향에서 피드백이 오므로 빠르게 캐치할 수 있다. 시선이 이미 왼쪽에 가있기 때문.
- **공간의 일관성**: 이 배치를 바꾸면 사용자가 의식하지 못하더라도 뭔가 어색하다고 느낀다. 건드리지 말 것.

소셜 연출 기술 세부:
- **말풍선 슬롯**: flush/capped → "flush" 슬롯, 채팅 → "chat" 슬롯. 같은 사람의 두 메시지 공존 가능
- **연타 흡수**: 같은 사람이 너무 빠르게 연속 전송하면 앞엣것을 버리고 최신 1건만 렌더 → 말풍선 도배/깜빡임 방지
- **방향 일관성**: 사람마다 좌/우가 고정됨. 계속 바뀌면 옆 칸 사람이 왔다갔다 하는 느낌이라 이상함. 같은 사람의 채팅과 물내림 멘트가 모두 같은 방향(같은 칸)에서 나오도록 보정되어 있다.

## 채팅 — 도배 당해도 답답하지 않은 UX

차단됐다는 사실을 부드럽게, 흐름을 끊지 않고 전달한다.

- **도배 판정**: 3초 안에 N번 연타 OR 같은 문장/유사 웃음 반복
- **차단**: 입력창 + 전송 버튼 10초 잠금 (팝업 없음, 화면 전환 없음)
- **카운트다운**: 인풋 placeholder에서 "N초 후 채팅 가능" 텍스트로 표시. 시간 지나면 자동 복원.
- **서버 연타 방어**: 공유/자랑 클릭은 800ms 안의 연타를 JS 단에서 삼킴 (서버로 안 보냄)

## 이스터에그 — 체류 시간 자연 연장

사용자가 "이거 뭐야?" 하며 발견하게 만들어야 한다. 찾으라고 안내하면 이스터에그가 아님.

| 이스터에그 | 트리거 | 반응 |
|-----------|--------|------|
| 천장 형광등 | 10초 안에 6번 빠르게 탭 | 불 깜빡임 + 옆 칸 말풍선 |
| 문 잠금 손잡이 | 10초 안에 6번 탭 | 걸쇠 슬라이드 애니메이션 + 옆 칸 반응 |
| 휴지 걸이 | 탭 1회 | 한 장 뜯겨 바닥 낙하 + 몇 초 후 재충전 |

각 이스터에그는 쿨다운이 있어 연속 트리거를 방지한다.

## 모바일 UX — 손가락 사용감까지

99%가 모바일. 엄지 하나로 모든 걸 조작한다는 전제로 설계한다.

- **탭 하이라이트 제거**: `-webkit-tap-highlight-color: transparent` → 파란 플래시 없음
- **텍스트 선택 방지**: `user-select: none` → 오래 눌러도 텍스트 선택 안 됨
- **탄성 스크롤 방지**: `overscroll-behavior-y: contain` → iOS 튕김 없이 앱처럼
- **레이아웃 시프트 0**: 숨겨진 요소는 항상 `opacity:0`으로 공간 확보 후 페이드인. `display:none` → 갑자기 등장 금지
- **탭 영역 확장**: 이스터에그 등 작은 요소는 투명 오버레이로 실제 탭 영역을 아이콘보다 크게

## 낙관적 업데이트 & 이벤트 버스

React 컴포넌트(PayslipModal)는 소켓에 직접 접근하지 않는다.
게임 엔진(game.js)이 소켓을 소유하고 `APP_EVENTS`로 통신한다.

- `payslipOpen`: 물내림 완료 시 game.js → PayslipModal (모달 열기)
- `payslipStamped`: 도장 찍힌 직후 PayslipModal → game.js (영수증 버튼 reveal)
- `brag`: 자랑하기 클릭 즉시(`await` 이전) game.js → socket emit
  → 공유 시트가 떴다가 취소돼도 집계 유실 없음
- 물내림 자체도 낙관적: 클릭 즉시 UI 반영(금액·컨페티·멘트), 소켓 emit은 그 다음

## 연결 UX — 끊겨도 당황하지 않게

```
연결 시도 → (1.5초 경과 시) "···" 로딩 표시
연결 됨   → 실제 인원 수
짧은 블립 → 아무것도 안 바뀜 (깜빡임 방지)
1.5초 경과 → "···" 재등장
8초 이상 실패 → "연결 끊김" (최후 수단)
```

재연결은 Socket.io 자동 백오프 (800ms → 최대 5000ms).
오프라인 중에도 타이머·적립은 로컬에서 계속 동작한다.

## 서버 최적화 — $5 안에서 전부

- **`pushAdminLive()` 디바운스**: 어드민 데이터는 변동 시에만 묶어서 push. 매 이벤트마다 쏘지 않음.
- **드리프트 인터벌**: `driftDelay()` 가중 랜덤 (10초~10분). 정기 setInterval 없음.
- **Redis 쓰기 최소화**: flush 시에만 씀. 1초 틱마다 쓰지 않음.
- **어드민 소켓 분리**: `adminRoom`으로만 push → 일반 사용자에게 admin 데이터 노출 없음.
- **가짜 소켓**: `NEXT_PUBLIC_RT !== "1"` 이면 로컬에서 서버 없이 동작.

## 핵심 시스템 설계

### 접속자 수 드리프트 (`server/lib/keys.js`)

**목적**: 혼자 들어와도 다른 사람들도 있다는 느낌.

**절대 하면 안 되는 것**: N초마다 정확히 +1 / 항상 최대값 수렴 / 시간대 무관 패턴

**설계**:
- `WD`(평일)/`WE`(주말) 배열: 24시간 시간대별 `[ceilRatio, biasUp]`
- `ceilRatio`: 설정값 대비 실효 상한 비율 (새벽 2시 = 0.20, 오후 2시 = 1.00)
- `biasUp`: 올라갈 확률 (점심 12시 = 0.28 강한 하락 / 오전 10시 = 0.58 상승 편향)
- 드리프트 인터벌: 15%→10~30초 / 25%→30~90초 / 25%→90~180초 / 20%→180~360초 / 15%→360~600초
- 이동 패턴: 6% 점프 / 15% 큰이동(±2~4) / 79% ±1
- 표시: `presence.size + presenceFloorNow` (덧셈, 어드민은 실제값만)

**체감 (max=10 기준)**:
- 오전 10시: 5→7→8→9→10→9 (상승, 중간에 꺾임)
- 점심 12시: 9→6→3→2 (확 빠짐)
- 새벽 2시: 1→2→1 (긴 정적)
- 10에 도달 못 하는 날도 있음. 매일 틀은 같지만 경로는 다름.

### 공유 페이지 슬롯머신 (`LiveEarningsBanner.tsx`)

**목적**: 지금 이 순간에도 돈이 쌓이고 있다는 느낌.

**절대 하면 안 되는 것**: N초마다 일정 금액 / N등분 / 실제값에 완전 도달

**설계**:
- 시작: 실제 누적의 **95%**에서 출발 (5% 여유 갭) — 초반 점프가 커서 눈이 피로한 문제 해소
- 스텝: **남은 갭의 35~55%** — 기하급수 수렴. 10번이면 갭의 0.3% 미만 잔존 ≈ 사실상 최대치.
- 롤링: 1.2~1.8초 (ease-out cubic)
- 쿨타임: 0.3~0.7초 (균일 범위) — 롤 1사이클 ~2초 × 10번 = **20초에 최대치 근접**
- 초기 대기: 3.5초 (영수증 먼저 읽을 충분한 여유 후 슬롯 시작)
- **20초 체류**: 10번 롤 완료, 잔여 갭 사실상 0. 이후 움직임 없음.
- 절대 real에 완전 도달하지 않음 (기하급수 수렴의 수학적 특성)

## 어드민

- `presenceFloorMax` (0~99): 드리프트 최대 추가 인원. 기본값 10.
- `presenceFloorAuto` (on/off): 드리프트 자동 여부
- 설정 저장 즉시 → `resyncPresenceFloor()` + `broadcastPresence()` + `pushAdminLive()` 연동
- 어드민은 항상 실제 presence만 보고, 드리프트 floor는 별도 표시

## 절대 하면 안 되는 것

1. **기계적 주기**: "N초마다", "N명씩 정확히" — 무조건 금지
2. **단순 N등분**: 갭을 N칸으로 똑같이 쪼개서 일정 간격으로 올리는 방식 → 기계 티 남. (슬롯머신은 "남은 갭의 N%씩" 기하급수 수렴 방식을 사용하는데, 이는 10초 안에 96% 근접 + 20초면 사실상 정지라는 의도된 설계다 — 혼동 주의)
3. **시간대 무시**: 새벽 2시에 10명이 보이면 안 됨
4. **완전 수렴**: 숫자가 상한에 딱 맞아 멈추면 가짜 티 남
5. **티어/상위N%**: 이미 제거됨. 절대 되살리지 말 것
6. **어드민에게 드리프트 적용**: 운영자는 실제 숫자를 봐야 함
7. **즉시 전부 노출**: 단계적 등장이 원칙. 처음부터 다 보이면 노이즈
8. **명시적 CTA**: "공유하세요!", "눌러보세요!" 같은 직접 문구 금지
9. **레이아웃 시프트**: `display:none` → 갑자기 등장 금지. `opacity:0`으로 공간 확보 후 페이드인
10. **Redis/서버 과부하**: 틱마다 Redis 쓰기, 매 이벤트마다 admin push — 반드시 디바운스/배치
11. **흐름 끊기**: 경고·차단은 현재 화면 안에서. 팝업·모달로 흐름을 자르지 말 것
12. **설계 변경 시 이 파일 미수정**: 기능/UX 변경이 있으면 반드시 이 파일도 함께 업데이트할 것

## AI 협업 원칙 — 의견을 물을 때

운영자가 "이게 맞나?", "이렇게 하는 게 낫나?", "어떻게 생각해?" 처럼 판단을 요청할 때 지켜야 할 원칙.

### 편향 없는 분석이 최우선이다

운영자의 말에 무조건 동의하는 것은 도움이 아니다. 반대로 습관적으로 반론을 제기하는 것도 마찬가지다.  
**AI가 편향된 피드백을 반복하면 운영자가 자신의 사고 틀에 매몰될 위험이 있다.** 이를 운영자 본인이 매우 경계한다.

기준은 항상 하나다: **"이 서비스를 처음 보는 일반 사용자가 어떻게 받아들일까?"**  
운영자의 시각이 아니라 타겟 페르소나(20~50대 직장인, 첫 방문, 모바일) 관점에서 판단한다.

### 의견 요청 시 답변 구조

1. **장점** — 이 방향이 맞다면 왜 맞는지 구체적으로
2. **단점 또는 리스크** — 놓치고 있는 시각, 발생할 수 있는 문제
3. **결론** — 장단점을 저울질한 뒤 가장 합리적이라고 판단되는 방향 하나를 명확하게 제시

결론 없이 "둘 다 방법이 있어요"로 끝내는 것은 금지. 판단을 위임받은 것이므로 명확한 입장을 내야 한다.

### 운영자가 틀렸을 때

동의하지 않으면 동의하지 않는다고 말한다.  
단, 감정 없이 — "이 부분은 이런 이유로 다르게 보입니다"처럼 근거를 붙인다.  
한 번 의견을 전달했으면 그 이상 반복하지 않는다. 운영자가 최종 결정권자다.

### 운영자가 맞았을 때

틀린 게 없으면 굳이 반론을 만들어내지 않는다.  
단, "맞습니다"만으로 끝내지 말고 왜 맞는지를 덧붙여 운영자가 확신을 가질 수 있게 한다.

### 개발 착수 전 코멘트 원칙

운영자가 "~~하는 게 좋을 것 같다", "~~를 하면 좋을 것 같다"처럼 제안형으로 말할 때:

**무조건 코멘트를 먼저 하고 개발을 시작한다.**

운영자는 결과를 보기 전에 방향을 바꾸거나 중단할 기회가 필요하다.  
코멘트 없이 바로 구현하면 운영자가 원치 않는 결과를 받아들고 나서야 수정을 요청해야 하는 상황이 생긴다.

코멘트에 담을 것:
1. 이 방향으로 구현한다는 확인
2. 구체적으로 무엇을 어떻게 할 건지 (1~3줄)
3. 주의해야 할 점이나 선택지가 있다면 명시

그 다음 운영자가 반응(승인·수정·중단)할 틈을 주고, 별도 제지가 없으면 구현 진행.
